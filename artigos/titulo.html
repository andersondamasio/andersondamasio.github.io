Desvendando o mistério da autorização na arquitetura de software: dentro do modelo de domínio ou na camada de aplicação?

Essa notícia me fez refletir sobre como o mercado ainda lida mal com a questão de onde a autorização deve residir em uma arquitetura de software. Muitas vezes, nos deparamos com a dúvida: a autorização deve estar dentro do modelo de domínio, especialmente se estamos utilizando o Domain-Driven Design (DDD)? Ou será que ela deve sempre permanecer na camada de aplicação?

Para responder a essas questões, é importante entender o papel da autorização em um sistema e como ela pode ser implementada de forma eficiente para garantir a segurança e a integridade dos dados.

Em termos gerais, a autorização refere-se ao controle de acesso a recursos e funcionalidades dentro de um sistema. Ela determina quem pode fazer o quê, com base em regras predefinidas. No entanto, a questão de onde essa lógica deve residir na arquitetura do software pode variar de acordo com o contexto e as necessidades específicas do projeto.

No contexto do Domain-Driven Design (DDD), a ideia principal é que a lógica de negócios e as regras do domínio devem estar contidas dentro do modelo de domínio. Isso inclui as regras de autorização que determinam quem pode executar determinadas ações dentro do sistema. Portanto, para muitos defensores do DDD, a autorização deve estar integrada ao modelo de domínio, garantindo que as regras de acesso sejam explicitamente definidas no núcleo da aplicação.

Por outro lado, há aqueles que argumentam que a autorização deve permanecer na camada de aplicação, separada do modelo de domínio. Essa abordagem permite uma maior flexibilidade e facilidade de manutenção, uma vez que as regras de autorização podem ser facilmente modificadas sem afetar diretamente o modelo de domínio. Além disso, ao manter a autorização na camada de aplicação, é possível centralizar a lógica de controle de acesso e garantir uma maior coesão e separação de responsabilidades dentro do sistema.

Na prática, a escolha entre integrar a autorização ao modelo de domínio ou mantê-la na camada de aplicação deve levar em consideração diversos fatores, como a complexidade das regras de autorização, a escalabilidade do sistema, a facilidade de manutenção e a clareza da arquitetura como um todo.

Para ilustrar essa discussão, vamos considerar um exemplo simples em C#:

public class PedidoService
{
    private readonly IAuthorizationService _authorizationService;

    public PedidoService(IAuthorizationService authorizationService)
    {
        _authorizationService = authorizationService;
    }

    public async Task CriarPedido(Pedido pedido)
    {
        if (await _authorizationService.AuthorizeAsync(pedido, "CriarPedidoPolicy"))
        {
            // Lógica para criar o pedido
        }
        else
        {
            throw new UnauthorizedAccessException("Usuário não autorizado a criar pedido.");
        }
    }
}


Nesse exemplo, a autorização é feita na camada de aplicação, utilizando um serviço de autorização externo. Isso permite uma maior flexibilidade e reutilização em diferentes partes do sistema, sem comprometer a integridade do modelo de domínio.

Em conclusão, a decisão de onde a autorização deve residir na arquitetura de software é uma escolha que deve levar em consideração diversos fatores e contextos específicos de cada projeto. Tanto a integração da autorização ao modelo de domínio quanto a sua manutenção na camada de aplicação têm vantagens e desvantagens, e cabe aos arquitetos e desenvolvedores avaliar qual abordagem melhor se adequa às necessidades do sistema em questão.
